# Cascade Observability Extension: Items 1-3

**Date:** 2026-03-31
**PR:** [#16592](https://github.com/datahub-project/datahub/pull/16592) (base — already merged cascade observability for graph-ref phase)
**Branch:** `feat/cascade-operation-observability`

## Problem

PR #16592 added `CascadeOperationContext` to the graph-ref scroll loop in `DeleteEntityService.deleteReferencesTo()`. However, two earlier phases of the same deletion — file reference cleanup and search reference cleanup — run **before** the cascade context opens. This creates three gaps:

1. **`deleteFileReferences`** (S3 object deletion + file entity soft-delete) has no aggregate metrics, no cascade correlation, and errors are logged but not counted.
2. **`deleteSearchReferences`** (scroll-based form/structured property reference cleanup) has no aggregate metrics, no cascade correlation, and per-asset errors are logged but not counted.
3. **Cascade ID propagation** — MCPs generated by `updateAspect()` in the graph-ref phase never set `systemMetadata`, so downstream Kafka consumers cannot correlate these MCPs back to the triggering delete operation.

## Decisions

| Decision                             | Choice                                                             | Rationale                                                                                                                                              |
| ------------------------------------ | ------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Single vs. per-phase cascade context | **Single context** covering all three phases                       | One operation ID for the entire deletion; simpler correlation                                                                                          |
| Estimated total at context creation  | **-1 (unknown)**                                                   | Total isn't known until each phase scrolls; estimated total is only used in a DEBUG log; actual `entitiesProcessed` in close() metrics is what matters |
| Cascade ID on `deleteAspect()` MCPs  | **Skip for now**                                                   | `EntityService.deleteAspect()` doesn't expose a SystemMetadata parameter; aspect deletions during reference cleanup are rare; follow-up item           |
| Error tracking in search-ref phase   | **`recordEntityProcessed()` per asset + `recordError()` in catch** | Matches the existing pattern in the graph-ref phase                                                                                                    |
| Approach                             | **Lift cascade context, thread through existing methods**          | Minimal API changes; follows existing patterns; no new classes                                                                                         |

## Design

### File changed

`metadata-service/services/src/main/java/com/linkedin/metadata/entity/DeleteEntityService.java`

No other files changed. No new files. No new dependencies.

### 1. Restructure `deleteReferencesTo()` (lines ~85-184)

Move `CascadeOperationContext.begin()` from line 136 (inside graph-ref phase only) to the top of the method, wrapping all three phases. Extract the dry-run path to exit early without creating a cascade context.

**Before:**

```
deleteFileReferences(...)            // no cascade
deleteSearchReferences(...)          // no cascade
scroll for relatedAspects preview
if (dryRun) return
CascadeOperationContext.begin(...)   // graph-ref only
  do { scroll + deleteReference }
```

**After:**

```
if (dryRun) {
  deleteFileReferences(..., null)
  deleteSearchReferences(..., null)
  scroll for relatedAspects preview
  return result
}

CascadeOperationContext.begin(..., -1)   // ALL phases
  deleteFileReferences(..., cascade)
  deleteSearchReferences(..., cascade)
  scroll for relatedAspects preview      // first batch for total count
  do { scroll + deleteReference }        // existing graph-ref loop (unchanged)
```

The existing graph-ref loop body is unchanged — it already calls `cascade.recordEntityProcessed()` and `handleError(..., cascade)`.

### 2. Changes to `deleteFileReferences()` (lines ~794-832)

Add `@Nullable CascadeOperationContext cascade` parameter.

In the `result.assets.forEach(...)` loop:

- After successful `deleteFileAndS3Object()`: call `cascade.recordEntityProcessed()` (guarded by `cascade != null`)
- In the catch block: call `cascade.recordError("file_delete_failed")` (guarded by `cascade != null`) alongside the existing `log.error()`

No changes to `deleteFileAndS3Object()` internals.

### 3. Changes to `deleteSearchReferences()` (lines ~589-611)

Add `@Nullable CascadeOperationContext cascade` parameter.

In the `assetsReferencingUrn.forEach(...)` loop:

- Before each `ingestProposal()` call: create `SystemMetadata` on the MCP if null, then call `cascade.attachToSystemMetadata(mcp.getSystemMetadata())` to stamp the cascade operation ID
- After processing each asset: call `cascade.recordEntityProcessed()`
- Both guarded by `cascade != null`
- Pass `cascade` through to `deleteSearchReferencesForAsset()` (see Section 4)

### 4. Changes to `deleteSearchReferencesForAsset()` (lines ~677-705)

Add `@Nullable CascadeOperationContext cascade` parameter.

In the per-aspect catch block (line 696-702): call `cascade.recordError("search_ref_update_failed")` alongside the existing `log.error()`.

### 5. Cascade ID propagation in `updateAspect()` (lines ~416-450)

Before calling `_entityService.ingestProposal()`:

```java
proposal.setSystemMetadata(new SystemMetadata());
cascade.attachToSystemMetadata(proposal.getSystemMetadata());
```

`updateAspect` is only called from within the graph-ref scroll loop where `cascade` is never null. No null guard needed.

### 6. Not changed

| Item                                      | Reason                                                                            |
| ----------------------------------------- | --------------------------------------------------------------------------------- |
| `CascadeOperationContext.java`            | No API changes needed                                                             |
| `deleteAspect()`                          | Can't propagate cascade ID without changing `EntityService` interface (follow-up) |
| `deleteFileAndS3Object()`                 | Per-file error handling stays as-is                                               |
| `PropertyDefinitionDeleteSideEffect.java` | Already instrumented, unchanged                                                   |
| Test files                                | Existing tests cover changed paths; public API unchanged                          |

## Metrics impact

The existing metrics now capture all three phases instead of just the graph-ref phase:

| Metric                               | Before                  | After                                                            |
| ------------------------------------ | ----------------------- | ---------------------------------------------------------------- |
| `datahub.cascade.duration`           | Graph-ref phase only    | Full deletion (files + search refs + graph refs)                 |
| `datahub.cascade.entities_processed` | Graph-ref entities only | Files + search-ref assets + graph-ref entities                   |
| `datahub.cascade.errors`             | Graph-ref errors only   | File delete errors + search-ref update errors + graph-ref errors |

The `operation_type` tag remains `"deleteReferencesTo"`. The `trigger_urn_type` tag is extracted from the deleted entity's URN (bounded cardinality).

New error types added: `file_delete_failed`, `search_ref_update_failed` (alongside existing `clone_failed`, `aspect_delete_failed`, `mcp_processor_failed`).

## Cascade ID propagation coverage

| MCP generation point                   | Cascade ID attached?                                                                                           |
| -------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| `updateAspect()` (graph-ref phase)     | **Yes** (this design)                                                                                          |
| `deleteSearchReferences` ingest calls  | **Yes** (this design)                                                                                          |
| `deleteFileReferences` soft-delete MCP | **No** — `deleteFileAndS3Object` builds its own MCP internally; would need refactoring to pass cascade through |
| `deleteAspect()` (graph-ref phase)     | **No** — `EntityService.deleteAspect()` doesn't accept SystemMetadata (follow-up)                              |
| `PropertyDefinitionDeleteSideEffect`   | **Yes** (already done in PR #16592)                                                                            |
